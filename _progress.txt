### Deviations / decisions

- **Picker is rule-only, not Step-B-entire.** The plan's Phase 1 step 4
  reads as if `RecurrencePicker` owned the whole of Step B (series name,
  advanced settings, etc.). The component-inventory row for the picker
  says it's shared with `RecurrenceEditDialog`, which doesn't need
  series-level settings. To make the picker cleanly reusable, Step B's
  shell (series name/description + advanced auto_publish /
  generation_window_weeks disclosure) lives in `RecurringEventWizard`
  itself, and the picker handles only the `RecurrenceRule*` subset.
  This matches how the edit dialog (Phase 3.2) will want to consume it.
- **No non-null assertions.** Followed scoped lint strictness (`--max-
  warnings 0` on touched files) and replaced the EventWizard-style `!`
  post-fix with explicit presence guards before the mutation fires
  (validation already runs before submit, so the guard is defensive only).
- **Auto-derive weekday hint.** When the user picks a `start` in Step A
  and the recurrence is weekly with no weekday selected, the wizard
  pre-selects the weekday of `start` as a convenience. This hinges on
  `seriesNameTouched` / unselected-weekday checks so it never
  overwrites user intent.
