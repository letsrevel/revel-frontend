## 2026-04-24 — Phase 3.4 GenerateNowButton landed

Shippable: the manual-generate affordance wired through the existing
`SeriesHeaderCard.onGenerateNow` prop (desktop action row + mobile
action sheet both covered by Phase 2.2/2.9). The dialog respects the
UX-doc §6 toast split (plural "{n} new occurrences scheduled through
{date}" vs. "Already up to date — no new occurrences to generate.")
and uses the Phase-0-seeded `successToast.one/other` + `upToDateToast`
keys verbatim.

### What shipped

- **`src/lib/components/event-series/admin/GenerateNowButton.svelte`**
  (195 lines — smaller than CancelOccurrenceDialog, as predicted).
    - bits-ui `Dialog` in the same pattern as CancelOccurrenceDialog:
      single-step, destructive-adjacent but not red-styled (generate
      is safe-by-default). Block-form eslint-disable around the
      bindable destructure for consistency with 3.3.
    - Body: a single `<input type="datetime-local">` for the optional
      `until` override + a helper line that surfaces the series'
      `generation_window_weeks` via an ICU `{weeks}` param. Empty input
      = use the default window (SDK accepts `body: null`).
    - Submit path branches toast on the response array:
      `response.length === 0` → `upToDateToast`; `=== 1` → `successToast.one`;
      `> 1` → `successToast.other` with `{ n, date }`. The `{date}` is
      the latest `start` among the freshly-generated events — defensive
      max in case the backend ever changes its ordering (currently
      creation order = ascending, but the max is the honest horizon).
    - POST via `organizationadminrecurringeventsGenerateEvents` with
      `{ until: new Date(localValue).toISOString() }` when the user
      supplies one (browser tz → UTC conversion), or `null` otherwise.
      Full `invalidateSeries` on success regardless of result size —
      even a zero-event response means the backend touched the horizon
      cursor, and `last_generated_until` on the admin detail surface
      may have moved.
    - `data-testid`s across the interactive surface:
      `generate-now-dialog`, `generate-now-until`, `generate-now-error`,
      `generate-now-cancel`, `generate-now-submit` — ready for Playwright
      at 3.7/3.8.

- **Dashboard wire-up** at the series dashboard page: imported
  `GenerateNowButton`, added `showGenerateNow = $state(false)`, threaded
  `onGenerateNow={() => (showGenerateNow = true)}` on the
  `SeriesHeaderCard` prop surface, mounted the dialog inside the
  `{#if series}` block next to the other Phase-3 dialogs.

- **i18n** — extended the existing Phase-0 `recurringEvents.generateNow.*`
  block (5 seeded keys: `untilLabel`, `successToast.one`,
  `successToast.other`, `upToDateToast`, `action`) with 6 more leaf keys:
  `title`, `description`, `untilHelper` (ICU `{weeks}` param),
  `submitting`, `cancelButton`, `errorToast`. Mirrored across EN/IT/DE
  with EN copy. `pnpm paraglide:compile` + `pnpm i18n:validate` →
  **4698 keys, 100% parity**.

### Deviations / decisions

- **Dialog over Popover.** Plan §Phase 3 §4 says "popover for the
  optional `until`". A dialog is one extra click but matches the rest
  of the Phase-3 pattern (SeriesHeaderCard's `onGenerateNow` was wired
  as a no-arg callback in Phase 2, same as the other header actions —
  switching to a popover would have required a separate trigger on the
  desktop row + a coerced bottom-sheet path on mobile; a dialog lets
  both paths share the same open mechanism). Net a11y + maintenance
  win vs. a single extra dismiss click. Filename kept as
  `GenerateNowButton.svelte` per the plan's component inventory.
- **Kept the datetime-local input free-form** (unlike the header-mode
  picker in CancelOccurrenceDialog, which uses a `<select>` over the
  upcoming occurrences list). Reason: `until` in generate has a
  different contract from `occurrence_date` in cancel — `until` is a
  horizon cap and any datetime the user picks maps onto the recurrence
  rule deterministically; there's no exact-match requirement to trap
  the user into 422s. Free-form is the right UX here.
- **Sequential `if/else if/else` rather than a single ICU
  `successToast` with `{n}`.** Paraglide's compiled output flattens
  the nested one/other into separate accessors
  (`.successToast.one` / `.successToast.other`), so the consumer has
  to branch. Matches how `CadenceDriftBanner` already consumes
  `drift.banner.title.one/other`. The extra `events.length === 0`
  branch falls out naturally to the `upToDateToast` key.
- **Full invalidation even on zero-event success.** `last_generated_until`
  on the admin detail can move forward without materialising new
  events (if the horizon was already past the window), and drift
  recomputes off the current rule + last-generated-until pair. A
  targeted invalidation would miss the horizon-advance case, so the
  uniform `invalidateSeries` call is the cheapest safe option.
- **No Vitest component test.** Same bits-ui Vitest harness carryover
  that has blocked every Phase-3 dialog so far. Playwright coverage
  lands at 3.7/3.8.

## 2026-04-24 — Phase 3.3 CancelOccurrenceDialog landed

Shippable: the cancel-occurrence dialog wired into both entry points
(row-level quick action + header-level picker), backed by a single
`CancelOccurrenceDialog.svelte` that branches on an optional
`initialDate` prop. Full series invalidation on success means the
dashboard's `ExdatesChipList`, drift banner, and occurrence rows all
refresh automatically — the backend flips `status=CANCELLED` on a
materialised event and appends to `exdates`; the FE just follows.

### What shipped

- **`src/lib/components/event-series/admin/CancelOccurrenceDialog.svelte`**
  (243 lines — the smallest Phase-3 dialog so far, as expected).
    - bits-ui `Dialog` wrapper mirroring the `RecurrenceEditDialog` /
      `TemplateEditDialog` pattern. No second step; the verbatim UX §5
      tickets-and-refund copy sits inline in a warning banner below
      the date context so the user sees it *before* confirming.
    - **Row mode** (`initialDate` is a non-empty ISO string): renders a
      read-only `<p>` with the formatted start under an "Occurrence to
      cancel" label. No picker. The backend requires an exact match
      against a scheduled start, so row mode re-uses `event.start`
      verbatim — no rounding, no re-formatting.
    - **Header mode** (`initialDate` is `null`): renders a native
      `<select>` populated from `occurrences.filter(status ∈ {draft,
      open})`. Preferred over a `datetime-local` input because the
      exact-match backend contract makes a free-form picker a 422-trap.
      The filter mirrors `OccurrenceRow`'s `isCancellable` gate. Empty
      state (no cancellable upcoming occurrences) renders a dashed-
      border helper ("No upcoming occurrences to cancel.") so the
      dialog is still navigable with keyboard.
    - Button row: **Keep date** (outline) + **Cancel date** (destructive
      variant). Destructive confirm flips to a loader + "Cancelling…"
      while the mutation is pending; `Keep` is disabled during pending
      and the dialog's `onOpenChange` close is also guarded — user
      can't dismiss a mid-flight POST.
    - POST via `organizationadminrecurringeventsCancelOccurrence` with
      `{ occurrence_date: selectedDate }`. On success: full
      `invalidateSeries(queryClient, slug, id)` which covers detail +
      drift + occurrences + the public series detail key. Toast copy
      is the Phase-0-seeded `successToast` ("Date cancelled."). On
      error: `response.error.detail` → banner + toast verbatim;
      `response.error.errors` → first message; else generic errorToast.
    - `data-testids` across the interactive surface:
      `cancel-occurrence-dialog`, `cancel-occurrence-selected-date`
      (row mode), `cancel-occurrence-picker`,
      `cancel-occurrence-picker-empty`, `cancel-occurrence-body`,
      `cancel-occurrence-error`, `cancel-occurrence-keep`,
      `cancel-occurrence-confirm` — lined up for the 3.7/3.8
      Playwright smoke.

- **Dashboard wire-up** at
  `src/routes/(auth)/org/[slug]/admin/event-series/[series_id]/+page.svelte`:
  imported `CancelOccurrenceDialog`, added `showCancelOccurrence` +
  `cancelOccurrenceInitialDate` state runes and two small helpers
  (`openCancelOccurrenceFromRow`, `openCancelOccurrenceFromHeader`)
  that flip `initialDate` before showing the dialog so the dialog's
  open-effect seed picks up the right source. Wired
  `SeriesHeaderCard.onCancelOccurrence` (header entry — was previously
  a no-op prop; the Phase-2 header card + action sheet already
  threaded it through with `data-testid="action-cancel-occurrence"`)
  and `OccurrenceRow.onCancelOccurrence` (row entry — was also a
  dangling prop, the row component had the callback + testid
  `row-cancel-occurrence` shipped in Phase 2.3). Dialog mounts inside
  the same `{#if series}` block as the other Phase-3 dialogs.

- **i18n** — extended the existing Phase-0
  `recurringEvents.cancelOccurrenceDialog.*` block (5 seeded keys:
  title, body, confirm, keep, successToast) with 7 more leaf keys:
  selectedDateLabel, picker.{label,placeholder,empty}, cancelling,
  errorToast. Mirrored across EN/IT/DE with EN copy per the Phase-0
  convention — real IT + DE translations land in Phase 5. `pnpm
  paraglide:compile` + `pnpm i18n:validate` → **4692 keys, 100%
  parity** across all three locales.

### Deviations / decisions

- **Single component + optional `initialDate` prop** rather than a
  `mode: 'row' | 'header'` discriminator. The shape is smaller (no
  enum plumbing, no `if (mode === …)` branches everywhere) and the
  truthiness check on `initialDate` reads the same either way.
- **`<select>` over `<input type="datetime-local">`** for header mode.
  UX doc §5 is explicit: "must be an exact match against a scheduled
  start". A datetime-local picker lets the user input *any* time and
  the backend would 422 any non-exact match — a trap we'd need to
  build client-side round-trip validation to avoid. A select of the
  existing upcoming starts is both safer (no mismatched timestamps)
  and faster for the organiser (click vs. full datetime entry).
- **No separate picker step.** The select + the warning body both fit
  in the same view; no reason to make the user click twice.
- **Eslint-disable became a block pair** (`eslint-disable` +
  `eslint-enable` around the destructure). 7 props pushed the
  destructure off the 100-char printWidth cap; the single-line
  `eslint-disable-next-line` pattern the other dialogs use only
  covers `let {`, not the per-identifier prefer-const fires on lines
  inside the multi-line destructure. Block pair keeps the rationale
  comment visible and shuts ESLint up on the whole statement. The
  autofixer's "unused directive" false positive (see 3.2 carryover)
  doesn't apply here because the block form has a different shape
  that the autofixer's static inference handles differently.
- **Full-series invalidation** on success via `invalidateSeries` — the
  plan §Phase 3 §3 says "invalidate series detail + occurrences +
  drift queries"; the helper already covers all three plus the
  public detail cache, which matches how cancelOccurrence's backend
  side-effects fan out (exdates bump in the recurrence detail, status
  flip on the materialised event, drift recomputation).
- **No dedicated 422 handling for an exact-match mismatch.** The
  `<select>` gates the payload to an exact scheduled start, so the
  422 path should be unreachable under the current flow. Backend
  error banners still surface `detail` verbatim if it ever happens
  (e.g. a stale query + an already-cancelled date in the select).
- **No Vitest component test for the dialog.** bits-ui Vitest harness
  tail is still broken (Vite 5↔6 realignment is the fix; see 2.10 /
  2.11 / 3.1 / 3.2 carryovers). Playwright coverage lands at 3.7/3.8.

## 2026-04-24 — Phase 3.2 RecurrenceEditDialog landed

Picked up straight from 3.1's clean slate. Shippable: the recurrence-edit
dialog wired into the series dashboard, the existing `RecurrencePicker`
extended with a proper read-only anchor notice, and the three-locale
`recurringEvents.recurrenceDialog.*` key block fleshed out for the new
surface.

### What shipped

- **`src/lib/components/event-series/admin/RecurrenceEditDialog.svelte`**
  (454 lines — RecurrencePicker does the heavy lifting so no sub-component
  split was necessary).
    - bits-ui `Dialog` wrapper mirroring the `TemplateEditDialog` pattern
      (same prefer-const eslint escape hatch on the bindable `open`
      destructure per TemplateEditDialog.svelte:36).
    - **Step 1 — form:** two sections. First is `RecurrencePicker`
      rendered with `dtstartReadOnly={true}` and fed the current rule
      (seeded from `series.recurrence_rule` on open). Second is a small
      "Series settings" section with the `auto_publish` toggle + the
      `generation_window_weeks` number input, reusing the wizard's
      existing `recurringEvents.wizard.autoPublish*` + `.generationWindow*`
      translations so no duplicate real copy had to land.
    - **Step 2 — confirm:** the verbatim UX-doc cadence-change warning
      ("Changing the schedule only affects occurrences not yet created.
      Upcoming occurrences already on the calendar keep their dates — you
      can cancel them individually if needed.") in a warning banner with
      `AlertTriangle` icon. Back + confirm buttons (`role="button"`,
      keyboard-accessible). The confirm step **only** renders when a
      recurrence-rule field actually changed — pure settings edits
      (`auto_publish` / `generation_window_weeks`) skip straight to PATCH
      since they don't shift already-scheduled dates and therefore don't
      warrant the "already on the calendar" warning.
    - **Dirty-tracking via `$derived.by`** — field-level compare against
      `series.recurrence_rule` + series-level settings. Each nullable
      field is normalized `undefined → null` before comparison so the
      picker's "switch frequency → clear weekdays/monthly_type/…"
      behaviour doesn't produce spurious diffs against backend nulls.
      Weekdays array compared by sorted-element-wise equality. The
      `EventSeriesRecurrenceUpdateSchema` body is built lazily from three
      separate `$derived` flags (`recurrenceChanged`, `autoPublishChanged`,
      `windowChanged`) so the "has changes" gate and the cadence-confirm
      branch both read from the same source of truth.
    - **Pre-submit validation** mirrors the wizard's `validateRecurrence`
      exactly (weekdays required on weekly, day_of_month on monthly-day,
      nth_weekday+weekday on monthly-weekday, mutualExclusionGuard for
      boundary, `until > dtstart`). Errors are surfaced via the picker's
      existing `validationErrors` prop — no duplicate UI needed.
    - **PATCH** via `organizationadminrecurringeventsUpdateRecurrence`
      with the minimal diff body. On success: `invalidateSeries(…)`
      covers the series detail + drift + occurrences triple — the
      backend recomputes drift after recurrence edits per PR #378, and
      the dashboard's `CadenceDriftBanner` refetches automatically.
    - **Error handling** per plan §Phase 1.6: `response.error.detail` →
      banner + toast verbatim; `response.error.errors` → first message
      surfaced. Close-while-pending guard on `onOpenChange` + the Cancel
      button matches TemplateEditDialog's pattern so a user can't dismiss
      a mid-flight PATCH.
    - **data-testids across every actionable surface**
      (`recurrence-edit-dialog`, `recurrence-edit-auto-publish`,
      `recurrence-edit-window`, `recurrence-edit-cancel`,
      `recurrence-edit-continue`, `recurrence-edit-confirm-banner`,
      `recurrence-edit-back`, `recurrence-edit-save`,
      `recurrence-edit-error`) + `recurrence-picker-anchor-readonly`
      + `recurrence-picker-anchor-value` on the picker for the 3.7
      Playwright smoke to drive the whole flow.

- **`src/lib/components/event-series/admin/RecurrencePicker.svelte`** —

### Deviations / decisions

- **`rule` / `autoPublish` / `generationWindowWeeks` initialized to
  neutral defaults, seeded from `series` in an `$effect`.** First pass
  seeded state directly from `series.X` at declaration time, matching
  the existing `SeriesSettingsDialog.svelte` pattern. That works at
  runtime but triggers Svelte 5's
  `svelte:state_referenced_locally` warning (prop-in-state captures
  only the initial value; the static analyzer can't see the re-seed
  effect). Switched to `$state({})` / `$state(false)` / `$state(8)` +
  a same-session open effect — drops the two new warnings introduced
  in the first pass, keeping the repo at its 339-warning baseline. The
  open effect gates on `if (!open) return` so first mount with
  `open=false` stays cheap and re-opens always pick up the freshest
  `series` snapshot.
- **Confirm step scoped to cadence changes only.** The UX doc's
  cadence-change copy ("Upcoming occurrences already on the calendar
  keep their dates…") is semantically a *cadence* warning — a pure
  settings edit (`auto_publish` toggle, window-weeks nudge) doesn't
  shift any scheduled dates and therefore doesn't merit the scare
  copy. The continue button's label flips too: "Review changes" when
  cadence is dirty, "Save changes" when only settings are. This keeps
  the confirmation friction proportional to the change's real impact
  without needing a second variant of the dialog.
- **Reused `recurringEvents.wizard.autoPublish*` +
  `.generationWindow*` keys** instead of duplicating them under
  `recurrenceDialog`. The wizard's copy is already well-tuned ("Auto-
  publish new dates" / "New dates go live the moment they are
  created." / "Occurrences are scheduled up to {weeks} weeks ahead…")
  and the dialog is editing the same two fields for the same series,
  so two copies of the same string would just be i18n churn.
- **No sub-component split needed.** TemplateEditDialog landed at 737
  lines and needed `TemplateLockedFields.svelte` extraction to pass
  the 750-line gate. RecurrenceEditDialog is 454 lines because
  `RecurrencePicker` owns the entire rule-editing surface — no
  extraction required for this one.
- **No Vitest component test for the dialog.** bits-ui Vitest harness
  tail is still broken (see 2.10 / 2.11 carryovers) — any component
  test transitively importing Dialog / DropdownMenu / Label runtime
  hits the `PartialEnvironment` Vite-6 proxy error. Playwright smoke
  is the viable coverage surface and lands at 3.7/3.8.
- **Full-series invalidation on success via `invalidateSeries`.** Plan
  §Phase 3 §2 says invalidate "series detail + drift queries". The
  `invalidateSeries` helper covers those plus the occurrences list,
  which is a superset we need here: a cadence change followed by a
  daily beat run could regenerate occurrences under the new cadence,
  and the dashboard's upcoming list will want to reflect any new
  drift-state derivations. One helper call keeps the mutation site
  free of bookkeeping.

## 2026-04-23 — Phase 2.11 Playwright smoke green (end of Phase 2)

Short continuation right after 2.10 was drafted. Goal was to actually
*run* the Playwright smoke end-to-end against the local stack and clear
the Phase-2 §QA gate. Ended up also surfacing a real dev-mode bug I
introduced in 2.10 (resolve.conditions wiping Vite defaults) and a pair

### What shipped

- **`vite.config.ts` — fixed the 2.10 regression.** The earlier
  `resolve.conditions: process.env.VITEST ? ['browser', 'svelte'] : []`
  looked defensive but actually broke prod/dev: setting `conditions` to
  `[]` wipes Vite's defaults (`module`, `browser`, `development`,
  `default`) so client-side SSR/hydration couldn't resolve nested
  package imports. The login page rendered but `appStore.fetchBackend
  Version` never fired — root `onMount` was throwing `lifecycle_outside
  _component` because module resolution crashed earlier in the chain.
  Rewrote to only *set* `resolve` when `VITEST` is truthy; when unset,
  leave it `undefined` so Vite keeps its defaults:

  ```ts
  const viteResolve = process.env.VITEST
      ? { conditions: ['browser', 'svelte'] }
      : undefined;
  // …
  resolve: viteResolve,
  ```

  Verified: Vitest harness still passes the Phase-2 component tests
  (same 49/49 green as 2.10); dev server now hydrates correctly; login
  page's demo dropdown renders with `BE v1.49.0` as expected.

- **`playwright.config.ts` — `reuseExistingServer: !process.env.CI`**
  added to the webServer block. Without it Playwright refuses to start
  when a dev server is already on :5173 (which is the ergonomic local
  loop: run `pnpm dev` in one terminal, drive Playwright from another).
  With it, Playwright reuses an existing server locally and still boots
  a fresh one under CI. Defaults match the repo's existing pattern for
  dev/CI parity.

- **`tests/e2e/recurring-series-dashboard.spec.ts` — made the spec
  actually pass:**
    - `waitForURL(/\/dashboard\//)` regex was mis-anchored (required a

      `/^continue$/i`.
    - Added a `waitForLoadState('networkidle')` + 2.5s settle after
      `/new-recurring` page load. Dev-mode paraglide compile + the
      auth-store init in the root layout can race early clicks; settling
      up-front is deterministic.
    - Post-create `waitForURL` was too loose
      (`/event-series/[^/?#]+/?$`) and matched the wizard URL itself
      (`/new-recurring`). Tightened to a literal UUID regex.
    - Dashboard header assertion hit two `h1`s (admin layout's org name
      + SeriesHeaderCard's series name) — disambiguated with

### Deviations / decisions

- **Scoped `resolve.conditions` to `VITEST` only.** The pitch was right
  (bits-ui needs the `svelte` condition) but the implementation was
  wrong — a non-test default of `[]` turns off Vite's built-in

- **`--workers=1` for the full matrix run.** Parallel workers hit the

### Deviations / decisions

- **Public GET for media + tags** rather than extending the admin schema.
  `EventSeriesRecurrenceDetailSchema` (the admin shape the dashboard
  already has) exposes only recurrence/template/exdates fields — it does
  NOT carry `logo` / `cover_art` / `tags`. Two options: (a) extend the
  backend schema, (b) fetch the public `EventSeriesRetrieveSchema` from
  inside the dialog on open. Went with (b): lazy (only runs while
  `open===true`), cached under `seriesQueryKeys.detail(seriesId)`, and
  invalidated by the existing `invalidateSeries` helper alongside the
  admin detail and drift. Zero backend churn. If the backend later folds
  media into the admin schema, the dialog can drop the extra query in one
  edit — the prop contract doesn't change.

  wire. Left slug alone; if a slug editor is needed later, either a
  backend schema change or a separate endpoint is required. Noted in the
  progress log so the next session doesn't look for it.

- **Upload-on-select + delete-on-click, not batched.** Media + tags each
  mutate immediately on interaction; only the basic-details form waits
  for an explicit Save. This matches the old `/edit` page's UX (no
  regression) and avoids the complexity of rolling back a partially-
  failed batch. The Close button guards against closing mid-mutation.

- **`prefer-const` eslint warning suppressed on the `$props()` destructure
  line only.** Bindable props force `let` for the whole destructure (same
  shape as DuplicateEventModal, which has accumulated the same warnings as
  part of the repo's ~889 pre-existing warning count). Used an inline
  `eslint-disable-next-line prefer-const` with an explanatory comment —
  avoids the false positive without hiding anything systemic.

- **ImageUploader reset after upload** — deferred. After a successful
  upload, the `ImageUploader`'s internal `selectedFilePreview` (data URI
  of the just-chosen file) keeps showing instead of the new server-hosted
  URL, because the component prioritises `selectedFilePreview` over the
  parent's `preview` prop. Not a regression (old `/edit` page had the
  same behaviour) and closing/reopening the dialog resets it. Can be
  addressed if Phase 5 polish asks for a post-upload visual refresh.

  - The render is a minimal but correct scaffold covering the shape the

    - Exdates chip list stub (Phase 2.4 swaps for the full component with
      the "Create one-off event for this date" action).
  - Templates are filtered client-side from the occurrence list
    (`!e.is_template`) — defence-in-depth on top of the backend's own
    filter.

### Deviations / decisions

- **No `eventseriesGetEvents`.** Plan §Phase 2 named it; the SDK doesn't
  export that name. Used `eventpublicdiscoveryListEvents({ event_series })`
  instead — it's the actual listing endpoint and it returns
  `EventInListSchema[]` with `is_template`, `is_modified`,
  `occurrence_index` inlined per row. No functional gap.
- **`Promise.allSettled`, not `Promise.all`.** Lets drift / upcoming
  failures degrade gracefully (logs + empty fallback) without blocking
  the page. Detail failure is still fatal — the dashboard can't render
  without the series shape.
- **TanStack Query usage matches Svelte 5 pattern in this repo** —
  `queryObject.data` (reactive proxy), not `$queryObject.data` (store).
  Followed the venue admin page's convention.
- **Back-nav** from the dashboard goes to
  `/org/{slug}/admin/event-series` (the list). Matches the sibling
  `new-recurring` page.
- **List-row click redirect** in the series list page and the `/edit`
  → `?settings=open` redirect both still deferred to Phase 2.8 — those
  need the settings dialog (2.7) to land first, otherwise the redirect
  has nothing to open.
